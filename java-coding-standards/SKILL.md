# Java编码规范检查 Skill

## 描述

基于《阿里巴巴Java开发手册（嵩山版）》的代码规范检查助手。帮助开发者检查Java代码是否符合规范，提供正例和反例参考，解答编码规范相关问题。

## 使用场景

- 代码审查时检查是否遵循规范
- 编写代码时咨询最佳实践
- 学习Java编码规范
- 解答编码规范相关问题

## 规范层级说明

- **【强制】**：必须遵守，违反可能导致严重问题
- **【推荐】**：建议遵守，有利于代码质量和可维护性
- **【参考】**：供参考，根据实际情况选择

## 核心规范要点

### 一、编程规约

#### (一) 命名风格

| 规则 | 级别 | 说明 |
|------|------|------|
| 代码命名不能以下划线或美元符号开始/结束 | 强制 | 如：_name / $Object |
| 严禁使用拼音与英文混合，禁止中文命名 | 强制 | 如：DaZhePromotion |
| 类名使用UpperCamelCase风格 | 强制 | 如：ForceCode / UserLoginController |
| 方法名、参数名、成员变量使用lowerCamelCase | 强制 | 如：localValue / getHttpMessage() |
| 常量名全大写，单词间用下划线隔开 | 强制 | 如：MAX_STOCK_COUNT |
| 抽象类以Abstract或Base开头 | 强制 | 如：AbstractService |
| 异常类以Exception结尾 | 强制 | 如：BusinessException |
| 测试类以要测试的类名+Test结尾 | 强制 | 如：UserServiceTest |
| 数组定义使用Type[]形式 | 强制 | 如：String[] args |
| POJO类布尔变量不加is前缀 | 强制 | 否则序列化会有问题 |
| 包名统一小写，点分隔符间仅一个自然语义单词 | 强制 | 如：com.alibaba.ai.util |
| 杜绝不规范缩写 | 强制 | 如：AbstractClass缩成AbsClass |
| 在常量与变量命名时，表示类型的名词放在词尾 | 参考 | 如：startTime / workQueue |
| 若模块、接口、类、方法使用设计模式，在命名中体现 | 推荐 | 如：OrderFactory / LoginProxy |
| 接口类中方法和属性不要加修饰符 | 参考 | 如：public abstract String f(); |
| 接口和实现类命名 | 强制 | 形容词为接口：AbstractTranslator；动词：ICar / CarImpl |
| 枚举类名带Enum后缀，成员全大写 | 强制 | 如：MemberTypeEnum / MALE |
| Service/DAO层方法命名规约 | 参考 | get/list/count/save/remove/update |
| 领域模型命名规约 | 参考 | DO/DTO/VO/AO/Query等 |

#### (二) 常量定义

| 规则 | 级别 | 说明 |
|------|------|------|
| 不允许魔法值直接出现在代码中 | 强制 | 如：String key = "Id#taobao_" + tradeId; |
| long或Long赋值时，使用大写L | 强制 | 如：Long a = 2l; (是小写l易混淆) |
| 不在一个常量类中维护所有常量 | 推荐 | 按功能归类，分开维护 |
| 常量复用层次：跨应用→应用内→子工程内→包内→类内 | 推荐 | 避免重复定义 |
| 禁止在变量前加final关键字定义常量 | 参考 | 不规范 |

#### (三) 代码格式

| 规则 | 级别 | 说明 |
|------|------|------|
| 大括号使用规范：左右大括号各占一行 | 强制 | 左大括号前不换行，后换行；右大括号前换行 |
| 左小括号和右边相邻字符间不空格 | 强制 | 如：if (空格a == b空格) |
| if/for/while/switch/do等保留字与括号间必须加空格 | 强制 | 如：while (true) |
| 任何二目、三目运算符左右都需加空格 | 强制 | 如：a + b |
| 采用4个空格缩进，禁止使用Tab | 强制 | IDE设置Tab为4空格 |
| 单行代码不超过120字符 | 强制 | 超过需换行 |
| 方法参数个数不超过5个 | 推荐 | 超过考虑封装对象 |
| 没有必要增加若干空行分隔不同逻辑 | 推荐 | 保持代码紧凑 |
| 单个方法总行数不超过80行 | 推荐 | 超过考虑拆分 |

#### (四) OOP规约

| 规则 | 级别 | 说明 |
|------|------|------|
| 避免通过对象引用访问静态变量/方法 | 强制 | 直接用类名访问 |
| 所有覆盖方法必须加@Override注解 | 强制 | 防止方法签名错误 |
| 可变参数必须放在参数列表最后 | 强制 | 如：public void method(String s, Integer... args) |
| 外部正在调用或依赖的接口，不允许修改方法签名 | 强制 | 避免接口调用方报错 |
| 不能使用过时类/方法 | 强制 | 应使用新API替代 |
| Object的equals方法容易抛空指针 | 强制 | 应使用常量或确定有值的对象调用 |
| 所有整型包装类对象比较用equals | 强制 | Integer缓存范围-128到127 |
| 浮点数比较用BigDecimal | 强制 | 禁止用==或equals比较浮点数 |
| BigDecimal禁止用double构造 | 强制 | 用String构造或valueOf |
| 基本数据类型与包装类使用标准 | 强制 | POJO类用包装类；RPC方法返回值用包装类 |
| 所有POJO类属性必须使用包装类 | 强制 | 数据库可能返回null |
| 序列化类新增属性时勿修改serialVersionUID | 强制 | 避免反序列化失败 |
| 构造方法禁止加入业务逻辑 | 强制 | 放在init方法中 |
| POJO类必须写toString方法 | 推荐 | 方便排查问题 |
| 禁止在POJO类中对属性x默认设值为true | 推荐 | setter方法中避免 |
| 类成员方法访问顺序 | 参考 | 类成员→构造方法→公有方法→私有方法 |
| setter方法中参数名与类成员变量名一致 | 参考 | this.字段 = 字段 |
| 循环体内使用append方式扩展字符串 | 强制 | 避免使用+连接 |
| final可提高程序响应效率 | 推荐 | 声明成final的情况：不需要改变的变量、参数、类等 |
| 慎用Object的clone方法 | 推荐 | 浅拷贝问题 |
| 类成员与方法访问控制从严 | 推荐 | 任何类、方法、参数、变量严控访问范围 |

#### (五) 日期时间

| 规则 | 级别 | 说明 |
|------|------|------|
| 日期格式化时使用y表示年 | 强制 | yyyy表示年，YYYY表示当天所在周属于哪年 |
| 获取当前毫秒数用System.currentTimeMillis() | 推荐 | 而非new Date().getTime() |
| 不允许程序中显式用Calendar进行日期计算 | 推荐 | 使用LocalDate或DateTimeFormatter |
| JDK8+使用Instant代替Date | 推荐 | LocalDateTime代替Calendar |
| 不允许在其他系统或项目中使用自定义日期枚举 | 参考 | 使用统一枚举或第三方包 |

#### (六) 集合处理

| 规则 | 级别 | 说明 |
|------|------|------|
| 关于hashCode和equals处理 | 强制 | 只要重写equals必须重写hashCode |
| ArrayList的subList结果不可强转ArrayList | 强制 | 返回的是内部类SubList |
| Map的key为自定义对象时，必须重写hashCode和equals | 强制 | 否则无法正确判断key是否存在 |
| 使用集合转数组必须用集合的toArray(T[] a) | 强制 | 传入类型完全一致、长度为0的数组 |
| 使用工具类Arrays.asList()把数组转换成集合 | 强制 | 不能使用修改集合相关方法 |
| 泛型通配符<? extends T> | 强制 | PECS原则：Producer Extends Consumer Super |
| 无泛型限定的集合赋值给泛型限定的引用会丢类型信息 | 强制 | 编译不报错，运行报错 |
| 不要在foreach循环里进行remove/add操作 | 强制 | 使用Iterator方式或removeIf |
| 集合初始化时指定初始值大小 | 强制 | 如：new ArrayList<>(100) |
| 使用entrySet遍历Map | 推荐 | 而非keySet遍历 |
| 高度注意Map类集合K/V能不能存储null值 | 参考 | HashMap可存null key/value，ConcurrentHashMap不允许 |

#### (七) 并发处理

| 规则 | 级别 | 说明 |
|------|------|------|
| 获取单例对象必须线程安全 | 强制 | 饿汉式/双重校验锁/静态内部类/枚举实现 |
| 创建线程或线程池时指定有意义的名称 | 强制 | 方便排查问题 |
| 线程资源必须通过线程池提供 | 强制 | 不允许显式创建线程 |
| 线程池不允许使用Executors创建 | 强制 | 应通过ThreadPoolExecutor创建 |
| SimpleDateFormat是线程不安全类 | 强制 | 使用ThreadLocal或JDK8的DateTimeFormatter |
| 必须回收自定义ThreadLocal变量 | 强制 | 在finally代码块中remove |
| 高并发场景同步调用应考虑锁性能 | 强制 | 能锁区块不锁方法体 |
| 对多个资源加锁须保持一致顺序 | 强制 | 避免死锁 |
| 加锁与解锁要成对出现 | 强制 | try-finally结构 |
| tryLock方式必须判断是否获取锁成功 | 强制 | lock.lock(); try{...} finally{ lock.unlock(); } |
| 并发修改同一记录避免更新丢失 | 强制 | 使用乐观锁version或悲观锁 |
| Timer运行多个TimeTask时，一个异常终止所有 | 强制 | 使用ScheduledExecutorService替代 |
| 资金相关金融敏感信息使用悲观锁 | 推荐 | 一锁、二判、三更新、四释放 |
| CountDownLatch异步转同步注意countDown | 推荐 | 确保子线程catch后能执行countDown |
| 避免Random实例被多线程使用 | 推荐 | 使用ThreadLocalRandom |
| volatile解决多线程内存不可见问题 | 推荐 | 但不能解决原子性问题 |
| HashMap在容量不够进行resize时 | 参考 | 高并发下可能出现死链 |
| ThreadLocal对象使用static修饰 | 参考 | 避免线程池环境下内存泄漏 |

#### (八) 控制语句

| 规则 | 级别 | 说明 |
|------|------|------|
| 在一个switch块内必须包含default语句 | 强制 | 放在最后 |
| switch块内每个case要么break，要么注释说明 | 强制 | 避免穿透 |
| 在if/else/for/while/do语句中必须使用大括号 | 强制 | 即使只有一行代码 |
| 三目运算符condition?表达式1:表达式2中 | 强制 | 表达式1和2类型必须一致 |
| 高并发场景避免使用"等于"判断中断 | 推荐 | 使用区间判断 |
| 表达异常分支用卫语句 | 推荐 | if (condition) { return; } |
| 避免取反逻辑运算 | 推荐 | 如：if (!(x >= 1)) → if (x < 1) |
| 业务上特殊场景可进行"快出"处理 | 参考 | 提前返回减少嵌套 |

#### (九) 注释规约

| 规则 | 级别 | 说明 |
|------|------|------|
| 类、类属性、类方法的注释必须使用Javadoc规范 | 强制 | /** 内容 */ 格式 |
| 所有抽象方法必须使用Javadoc注释 | 强制 | 包含参数、返回值、异常说明 |
| 所有的类都必须添加创建者和创建日期 | 强制 | @author @date |
| 方法内部单行注释用 //，多行用 /* */ | 强制 | 与代码对齐 |
| 所有枚举类型字段必须有注释 | 强制 | 说明每个枚举值的用途 |
| 与其"半吊子"注释不如不写 | 推荐 | 注释要准确表达代码意图 |
| 注释内容修改代码要同步更新 | 推荐 | 废弃注释及时删除 |
| 注释的双斜线与注释内容之间有且仅一个空格 | 参考 | // 注释内容 |
| 方法参数注释方式 | 参考 | 参数放在上行，与注释对齐 |
| TODO标记格式 | 推荐 | TODO (author) by date: action |

### 二、异常日志

#### (一) 错误码

| 规则 | 级别 | 说明 |
|------|------|------|
| 错误码不能直接输出给用户 | 强制 | 作为提示信息资源引用 |
| 错误码只做一层定义 | 强制 | 不分模块，便于维护 |
| 全局错误码分层管理 | 强制 | 一层为应用级，二层为模块级 |
| 错误码采用A-B-C格式 | 强制 | A:应用级 B:模块级 C:错误类别 |

#### (二) 异常处理

| 规则 | 级别 | 说明 |
|------|------|------|
| Java类库中可以通过throw抛异常的都要捕获 | 强制 | 如：NumberFormatException |
| 不允许用RuntimeException | 强制 | 定义业务异常继承RuntimeException |
| 捕获异常后不要不做任何处理 | 强制 | 至少记录日志 |
| 不允许使用Exception作为throws的异常类型 | 强制 | 应明确具体异常类型 |
| 方法返回值可以为null，不返回错误码 | 推荐 | 可封装为Result<T> |
| 防止NPE是程序员的基本修养 | 强制 | 注意数据库、远程调用、集合、对象属性等可能为null |
| 定义时区分unchecked/checked异常 | 推荐 | 业务异常用unchecked |
| 避免出现重复代码 | 推荐 | 将重复代码抽取到方法中 |

#### (三) 日志规约

| 规则 | 级别 | 说明 |
|------|------|------|
| 应用中不可直接使用日志系统Log4j/Logback | 强制 | 使用日志框架SLF4J |
| 日志文件至少保存15天 | 推荐 | 根据业务场景调整 |
| 应用中的扩展日志命名方式 | 推荐 | appName_logType_logName.log |
| 对trace/debug/info级别的日志输出 | 强制 | 先判断日志级别再输出 |
| 避免重复打印日志 | 推荐 | 浪费磁盘空间 |
| 生产环境禁止直接用System.out或System.err | 强制 | 使用日志框架 |
| 异常信息应该包括两类信息 | 强制 | 案发现场信息和异常堆栈信息 |
| 日志打印时禁止直接用JSON工具转换对象 | 推荐 | 可能因对象循环引用报错 |
| 谨慎地记录日志 | 推荐 | 避免输出大量无用日志 |
| 使用TraceId标识一次请求链路 | 推荐 | 便于日志追踪 |

### 三、单元测试

| 规则 | 级别 | 说明 |
|------|------|------|
| 好的单元测试必须遵守AIR原则 | 强制 | Automatic/Independent/Repeatable |
| 单元测试应该是全自动执行的 | 强制 | 非交互式 |
| 单元测试之间不允许互相调用 | 强制 | 保持独立性 |
| 单元测试必须可以重复执行 | 强制 | 不能受外界环境影响 |
| 对于单元测试要保证测试粒度足够小 | 强制 | 测试方法级 |
| 单元测试覆盖率要求 | 参考 | 核心业务需高覆盖率 |
| 核心业务代码必须写单元测试 | 推荐 | 保证代码质量 |
| 单元测试代码必须写在工程目录src/test/java | 强制 | 不允许写在src/main/java |
| 单元测试的基本目标 | 参考 | 语句覆盖70%以上，核心模块100% |
| 编写单元测试代码遵守BCDE原则 | 参考 | Border/Correct/Design/Error |
| 和数据库相关的单元测试 | 推荐 | 可以考虑使用内存数据库 |
| 对于不可测的代码进行重构 | 推荐 | 提高可测试性 |
| 在设计评审阶段根据测试难度评估设计合理性 | 推荐 | 测试驱动设计 |
| 不允许使用Random随机值做单元测试 | 推荐 | 应使用固定值 |
| 单元测试方法命名规范 | 推荐 | test_方法名_预期结果_输入条件 |

### 四、安全规约

| 规则 | 级别 | 说明 |
|------|------|------|
| 隶属于用户个人的页面或功能必须进行权限控制校验 | 强制 | 防止越权访问 |
| 用户敏感数据禁止直接展示 | 强制 | 必须进行脱敏处理 |
| 用户输入的SQL字符必须进行严格过滤 | 强制 | 防止SQL注入 |
| 用户请求传入的任何参数必须做有效性验证 | 强制 | 防止参数篡改 |
| 禁止向HTML页面输出未经安全过滤或转义的用户数据 | 强制 | 防止XSS攻击 |
| 表单、AJAX提交必须执行CSRF安全过滤 | 强制 | 验证Token |
| URL外部传入参数必须禁止直接使用 | 强制 | 必须进行安全校验 |
| 在使用平台资源时必须验证用户是否有权限使用 | 强制 | 如优惠券、红包 |
| 文件上传必须限制文件大小 | 强制 | 防止DoS攻击 |
| 禁止在日志中打印敏感数据 | 强制 | 如密码、身份证号 |
| 用户密码不允许明文存储 | 强制 | 必须加密存储 |

### 五、MySQL数据库

#### (一) 建表规约

| 规则 | 级别 | 说明 |
|------|------|------|
| 表达是与否概念的字段使用is_xxx命名 | 强制 | 数据类型unsigned tinyint |
| 表名、字段名必须使用小写字母或数字 | 强制 | 禁止数字开头，禁止两个下划线中间数字 |
| 表名不使用复数名词 | 强制 | 如：user而非users |
| 禁用保留字 | 强制 | 如：desc、range、match |
| 主键索引名为pk_字段名 | 强制 | 唯一索引uk_字段名，普通索引idx_字段名 |
| 小数使用decimal | 强制 | 禁止使用float和double |
| 字符串长度几乎相等用char | 强制 | 长度变化大用varchar |
| varchar长度不超过5000 | 强制 | 超过定义text，独立表用主键关联 |
| 表必备三字段 | 强制 | id, create_time, update_time |
| 表名使用业务名_模块名 | 强制 | 如：trade_order |
| 修改字段含义或新增字段 | 推荐 | 需更新字段注释 |
| 字段允许适当冗余 | 参考 | 高频访问、低修改频率字段 |
| 单表行数超过500万行 | 参考 | 考虑分库分表 |
| 字符集统一utf8mb4 | 强制 | 支持emoji |

#### (二) 索引规约

| 规则 | 级别 | 说明 |
|------|------|------|
| 业务上具有唯一特性的字段必须建成唯一索引 | 强制 | 即使是组合字段 |
| 超过三个表禁止join | 强制 | join字段数据类型必须一致 |
| varchar字段上建立索引必须指定索引长度 | 强制 | 根据区分度决定长度 |
| 页面搜索严禁左模糊或全模糊 | 强制 | 使用搜索引擎 |
| order by字段利用索引有序性 | 推荐 | 组合索引最后字段为排序字段 |
| 利用覆盖索引查询 | 推荐 | 避免回表 |
| 利用延迟关联优化超多分页 | 推荐 | 先定位id再关联 |
| SQL性能优化目标 | 推荐 | 至少range级，要求ref级 |
| 防止范围查询导致索引失效 | 强制 | 组合索引中范围查询后的索引失效 |
| 创建索引避免极端情况 | 参考 | 区分度不高、超长字段 |

#### (三) SQL语句

| 规则 | 级别 | 说明 |
|------|------|------|
| 不要使用count(列名)或count(常量) | 强制 | 使用count(*) |
| count(distinct col)计算该列除NULL外的行数 | 强制 | 注意NULL处理 |
| 某一列全为NULL时count(col)返回0 | 强制 | sum(col)返回NULL |
| 使用ISNULL()判断是否为NULL | 强制 | NULL!=NULL |
| 代码中写分页查询逻辑时禁止直接拼接参数 | 强制 | 防止SQL注入 |
| 禁止使用存储过程 | 强制 | 难以调试和扩展 |
| 禁止使用触发器 | 强制 | 业务逻辑应放入应用层 |
| in操作能避免则避免 | 推荐 | in后元素数量控制在1000个以内 |
| 禁止大表Join | 强制 | 注意性能影响 |
| 禁止使用NOT IN | 推荐 | 使用NOT EXISTS替代 |
| 事务中避免RPC调用 | 强制 | RPC不可回滚 |
| 禁止使用or连接条件 | 推荐 | 改为union或分次查询 |

#### (四) ORM映射

| 规则 | 级别 | 说明 |
|------|------|------|
| 表字段与Java属性命名映射 | 强制 | 下划线转驼峰 |
| 禁止使用*查询 | 强制 | 必须指定具体字段 |
| 禁止XML配置文件中参数用${} | 强制 | 使用#{}防止SQL注入 |
| POJO类属性与数据库字段一一对应 | 强制 | 不要添加非表字段 |
| 查询返回对象使用Map接收 | 推荐 | 或自定义DO/DTO |
| SQL更新操作使用DAO接口 | 强制 | 不允许通过Mapper XML |
| 不要写大而全的数据更新接口 | 强制 | 按需更新字段 |

## 使用方法

### 代码审查场景

当用户要求审查Java代码时：

1. 按照规范分类逐项检查
2. 指出违反规范的具体行号和代码
3. 说明违反的是【强制】/【推荐】/【参考】
4. 提供正例参考代码

### 咨询场景

当用户询问编码规范问题时：

1. 给出明确的规范要求
2. 说明规范级别
3. 解释原因（说明部分）
4. 提供正例和反例

### 学习场景

当用户学习Java编码规范时：

1. 可以按章节讲解
2. 结合实际代码示例
3. 说明规范的背景和意义
